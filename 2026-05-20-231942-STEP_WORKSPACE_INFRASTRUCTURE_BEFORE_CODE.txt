│                                                                              │
│ # SPEC.md: BIN-Based Routing Algorithm                                      │
│                                                                              │
│ ## Objective                                                                 │
│ Route Indian domestic RuPay cards to a domestic acquirer and international   │
│ cards to a global acquirer by evaluating the card's 6-digit Bank             │
│ Identification Number (BIN) through a configurable routing algorithm.        │
│                                                                              │
│ ## Architecture                                                              │
│ BIN routing is a new StaticRoutingAlgorithm::BinRouting variant that         │
│ participates in the normal routing pipeline — same level as Priority,        │
│ VolumeSplit, or Advanced. It is NOT a post-routing filter or override.       │
│                                                                              │
│ ### Data Model                                                               │
│ {                                                                            │
│   "type": "bin_routing",                                                     │
│   "data": {                                                                  │
│     "rules": [                                                               │
│       { "bin_prefix": "60",                                                  │
│         "connector": { "connector": "phonypay",                              │
│           "merchant_connector_id": "mca_xxx" } }                             │
│     ],                                                                       │
│     "default": { "connector": "stripe_test",                                 │
│       "merchant_connector_id": "mca_yyy" }                                   │
│   }                                                                          │
│ }                                                                            │
│                                                                              │
│ ### Flow                                                                     │
│ 1. Merchant creates a BinRouting algorithm via POST /routing                 │
│ 2. Algorithm is stored in DB as JSON blob, kind = advanced                   │
│ 3. On payment, perform_bin_routing() extracts card_bin from backend_input    │
│ 4. Matches against configured rules (prefix match, top-to-bottom)            │
│ 5. Returns matching RoutableConnectorChoice (with merchant_connector_id)     │
│ 6. Result flows through normal eligibility analysis                          │
│                                                                              │
│ ## Guardrails                                                                │
│ 1. PII Security: Only the 6-digit BIN prefix is logged                      │
│ 2. Fail-Open: Missing/short BIN returns default connector                    │
│ 3. Configurable: BIN mappings are API-driven, not hardcoded                  │
│ 4. Pipeline Integrity: Preserves merchant_connector_id                       │
│                                                                              │

│ # DECISIONS.md                                                               │
│                                                                              │
│ ## 1. Scope & Implementation Details                                        │
│ - Implemented BinRouting as a first-class StaticRoutingAlgorithm variant    │
│ - BIN-to-connector mappings are fully configurable via the API              │
│ - perform_bin_routing() extracts card_bin from backend_input.payment.card_bin│
│ - Logs only the BIN prefix slice via [BIN_ROUTER] — never the full card #   │
│                                                                              │
│ ## 2. Pragmatic Engineering Trade-offs                                       │
│ - Algorithm vs. Post-Routing Filter: Chose a new algorithm variant over a    │
│   post-routing connector reorder. A filter would discard the routing         │
│   pipeline's connector selection and lose merchant_connector_id.             │
│ - Reuse RoutingAlgorithmKind::Advanced: Mapping BinRouting to Advanced       │
│   avoids a PostgreSQL enum migration. The JSON "type" tag differentiates.    │
│ - Skip Decision Engine Sync: BinRouting is handled natively by Hyperswitch.  │
│   The DE doesn't understand BIN routing, so it's skipped during DE sync.    │
│                                                                              │
│ ## 3. Future Polish                                                          │
│ - Multiple BIN ranges: Add support for BIN range patterns                   │
│ - Card network detection: Auto-populate rules based on card network          │
│ - Admin UI: Visual editor for BIN routing rules in the merchant dashboard   │

  1. Payment arrives → make_dsl_input() extracts card_bin from card number

  4. Result flows through eligibility analysis (checks MCA validity)
  5. Connector executes the payment
